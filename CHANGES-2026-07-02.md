# Changes — 2026-07-02 session

Summary of what changed in this pass, and what's still open. Written so a
future session (yours or mine) doesn't have to re-derive this from git log.

## Fixed

- **Retry defect (network/retry.py, network/client.py)**: `retry()`
  previously only retried on raised exceptions; HTTP 5xx/429 responses
  (returned normally, not raised) silently bypassed retry entirely. Fixed
  centrally via a `should_retry` predicate now wired into both
  `HttpClient.get()` and the new `HttpClient.post()`. Removed the two
  duplicated manual retry loops that had been patched into
  `crossref.py`/`arxiv.py` as a workaround — they're no longer needed and
  the fix is now consistent across all 5 providers.
- **`execution/` package removed**: was fully orphaned (never imported
  anywhere) and its `BenchmarkRunner` class name collided with the real
  one in `benchmark/runner.py`.
- **`evaluation/registry.py` naming bug**: `"bsi"` was mistakenly
  registered to `BasicEvaluator` (copy-paste error), making `"basic"` and
  `"bsi"` silently identical. Fixed, and `BasicEvaluator` restored to its
  original distinct purpose (article-count smoke test over a raw
  `Dataset`).
- **`--provider mock` was unusable**: `MockProvider` was registered but
  had no parser wired into `PipelineRunner`, so it always raised
  `KeyError`. Added `parsers/mock.py` and wired it in — this is what makes
  fully offline testing (no network, no API keys) possible via
  `--generators mock`.

## Rebuilt (was the core architectural gap from the last review)

`BSIEvaluator` previously scored the *fetched article metadata itself*
(article count, presence of abstract/DOI, title length), mislabeled as the
seven BIO v1.0 dimensions — three of seven were literal duplicates
(d4=d1, d5=d2, d6=d3), and the total was an unweighted mean, not the
documented weighted `BSI_linear = sum(w_i * D_i)` formula.

It's rebuilt around the actual goal stated this session: **an engine that
automates BSI evaluation across multiple LLMs.** New pieces:

- `models/analysis.py`, `models/analyzed_article.py` — pairs a raw
  `Article` with a generated `Analysis` (the text an LLM produced).
- `evaluation/text_analysis.py` — offline, regex/keyword-based proxies
  (bilingual fa/en) for claim-tag extraction, layer detection, hedge vs.
  overclaim balance, conditional-reasoning density, implication density,
  domain breadth, lexical novelty, and a crude FACT-claim lexical
  grounding check. **These are heuristic proxies, not semantic
  understanding** — see the docstring at the top of that file for exactly
  what each one does and doesn't catch. Treat scores as first-pass
  triage, not certified BSI output, until validated against human
  judgement on real samples.
- `evaluation/bsi.py` — rewritten `BSIEvaluator`. Seven dimensions, each
  from an independent signal (no duplication), weighted with the real BIO
  v1.0 weights (0.22/0.18/0.18/0.17/0.12/0.08/0.05). Two extra diagnostic
  signals (`grounding_ratio`, `tag_coverage`) reported but not folded into
  the weighted total, since they don't map onto a single D_i.
- `generation/` — new package: `AnalysisGenerator` ABC +
  `AnthropicGenerator` / `OpenAIGenerator` (real API calls via the shared
  `HttpClient`, reading `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` from the
  environment — never hardcode a key) + `MockGenerator` (offline,
  deterministic, for tests and dry runs).
- `comparison/` — new package: `ComparisonSpec` (which generators, which
  prompt modes — e.g. `{"raw": ..., "bsi": ...}`), `CrossModelRunner`
  (orchestrates generate → evaluate over every article x generator x mode
  combination; one failed combination doesn't abort the run), plus
  markdown + JSON report output.
- `cli.py compare` — the actual automated-benchmark entry point:
  ```
  bsi-benchmark compare --provider crossref --query "..." \
      --generators anthropic,openai --bsi-prompt-file MASTER_PROMPT_BSI_v3.4.2.md \
      --output results/run1
  ```
  Note: `--bsi-prompt-file` is required and NOT fabricated by this tool —
  supply your actual BSI master prompt. See `examples/` for a runnable
  offline dry run with `--generators mock`.
- `common/registry.py` — a shared generic `Registry[T]`, used by the new
  `generation` module, so the pre-existing pattern of 5 near-identical
  hand-written registries (providers/parsers/evaluation/metrics/reporting)
  didn't get a sixth copy. Migrating the existing five to use this is a
  reasonable follow-up, not done here to limit blast radius.

## Verified

29 new/rewritten tests + 4 pre-existing offline-safe test files all pass
(38/40 in a manual runner here, since this sandbox has no network to
`pip install pytest`; the 2 "failures" are only missing the `tmp_path`
pytest fixture my manual runner doesn't provide — they're not real
failures and should pass under real `pytest`). Full end-to-end smoke test
of `compare` with `--generators mock` confirmed: fetch → generate (raw +
bsi modes) → evaluate → markdown/JSON export, all working, with the mock
"bsi" mode scoring meaningfully higher (0.828) than "raw" mode (0.205) —
which is the actual signal this benchmark exists to produce, now
automatic.

## Still open (not touched this session)

- `README.md` and `LICENSE` are still both empty (0 bytes) despite
  `pyproject.toml` claiming MIT and pointing `readme` at `README.md`.
- `settings.py` and `configuration/` are still orphaned (registered
  nowhere). Left alone since deleting them wasn't clearly safe without
  knowing if they're intended for a near-term config-loading feature —
  worth an explicit decision (wire them in, or remove) rather than a
  silent deletion.
- `metrics/`, `export/`, `history/` are tested in isolation but still not
  wired into the main `benchmark/runner.py` → `cli.py run` path.
- The lexical proxies in `text_analysis.py` have not been validated
  against real BSI output or human judgement — the saturation constants
  (`CONDITIONAL_SATURATION_PER_100_WORDS`, etc.) are rough guesses that
  should be recalibrated once real `compare` runs produce data.
- Only two LLM backends exist (Anthropic, OpenAI). Neither has been tested
  against the live API from this sandbox (no network here) — the request/
  response shapes are correct per each API's documented format but should
  be smoke-tested against the real endpoints in Termux before trusting a
  full benchmark run.
